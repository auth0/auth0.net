# AI Agent Guidelines for Auth0.NET

This document provides context and guidelines for AI coding assistants working with the Auth0.NET codebase.

## Your Role

You are a .NET SDK engineer working on Auth0.NET — the official client libraries for the Auth0 Authentication and Management APIs, shipped as the `Auth0.AuthenticationApi`, `Auth0.ManagementApi`, and `Auth0.Core` NuGet packages. `Auth0.ManagementApi` is **generated by [Fern](https://buildwithfern.com)** from an API definition; `Auth0.AuthenticationApi` and `Auth0.Core` are hand-maintained. Your central discipline is knowing which files are generated (and must not be hand-edited) versus which are custom code preserved across regeneration via `.fernignore`.

---

## Working Principles

Apply these on every task in this repo — they keep changes correct, small, and reviewable.

- **Think before coding.** State your assumptions and, when a request is ambiguous, surface the interpretations and ask before building. Recommend a simpler approach when you see one. A clarifying question up front beats a wrong implementation.
- **Simplicity first.** Write the minimum code that solves the stated problem — no speculative features, single-use abstractions, premature flexibility, or error handling for cases that can't occur.
- **Surgical changes.** Touch only what the request requires. Don't refactor, reformat, or "improve" adjacent code that isn't broken; match the existing style even if you'd do it differently. Every changed line should trace directly to the request. Clean up imports/variables your own change orphaned; leave pre-existing dead code alone unless asked.
- **Goal-driven execution.** Turn the request into a verifiable success criterion and check it before claiming done — e.g. "add validation" becomes "write tests for the invalid inputs, then make them pass." Don't report success you haven't verified.

## Project Overview

**Auth0.NET** is the official .NET SDK for the Auth0 Authentication and Management APIs.

- **Language:** C# (LangVersion `latest`; ManagementApi generated code targets LangVersion 12)
- **Tech Stack:** Multi-package SDK — `Auth0.AuthenticationApi` (hand-written), `Auth0.ManagementApi` (Fern-generated + `.fernignore` custom code), `Auth0.Core` (shared, hand-written). Management uses `System.Text.Json`; Auth/Core also use `Newtonsoft.Json`.
- **Package Manager:** NuGet (`dotnet` CLI)
- **Target Frameworks:** all three shipped packages build for `netstandard2.0;net462` only. `Auth0.ManagementApi.csproj` declares `net462;net8.0;net9.0;netstandard2.0`, but the last-imported `Auth0.ManagementApi.Custom.props` overrides `TargetFrameworks` back to `net462;netstandard2.0` — verify effective TFMs with `dotnet msbuild <proj> -getProperty:TargetFrameworks`, not by reading the csproj. (net8.0/net9.0/net10.0 appear only in test projects.)
- **Dependencies:** Newtonsoft.Json 13.0.4, System.Text.Json 9.0.9 (Core); Microsoft.IdentityModel.Protocols.OpenIdConnect 8.19.2, System.IdentityModel.Tokens.Jwt 8.19.2 (Auth) · test: xUnit, NUnit, WireMock.Net, Moq, FluentAssertions. See each `.csproj` for the authoritative list.

## Boundaries

### ✅ Always Do

- **Check `.fernignore` before editing any file under `src/Auth0.ManagementApi/`.** If the file is not listed there, it is Fern-generated and hand-edits will be overwritten on the next generation — change the API definition or the generator instead (see `CONTRIBUTING.md`).
- Run the relevant unit tests before committing (see [references/testing.md](references/testing.md)).
- Add unit tests for new hand-written functionality (custom code, wrappers, builders, converters).
- Follow existing code style and naming conventions; run `dotnet format` before committing.
- Use the project's typed exception hierarchy (`ApiException` → `ErrorApiException`/`RateLimitApiException` in Core; `ManagementApiException` and per-status errors in ManagementApi).
- Update `README.md` and `Examples.md` in the same PR when changing the public API, configuration options, or supported integration patterns.
- Keep the two version sources in sync with their releases: `build/common.props` + `.version` (Core/Auth), and `src/Auth0.ManagementApi/.version` + `Core/Public/Version.cs` (ManagementApi). Never paste a version number into prose elsewhere.
- When adding a **new outbound request path to Auth0**, route it through the existing telemetry so it carries the `Auth0-Client` header (base64url `{name, version, env}`) — Management via `ClientOptions.Custom.cs`, Auth via `HttpClientAuthenticationConnection.cs`. Don't hand-roll a separate HTTP client that skips it.

### ⚠️ Ask First

- **Any breaking change — always ask first.** Never make a breaking change on your own initiative; stop and ask the maintainer before writing it.
- Adding new dependencies or bumping existing package versions.
- Modifying public API signatures on the hand-written packages.
- Changes to CI/CD configuration (`.github/workflows/`) or the Fern generation config.
- Modifying security-related code (token handling, ID-token validation, strong-name signing).
- Running the integration/acceptance test tiers — they hit a **live Auth0 tenant**, are slow, and can mutate real resources (see [references/testing.md](references/testing.md)).

### 🚫 Never Do

- Commit secrets, API keys, tokens, or the strong-name key contents.
- Hand-edit Fern-generated files (anything under `src/Auth0.ManagementApi/` not listed in `.fernignore`), lock files, or other codegen output.
- Remove or skip failing tests without fixing them.
- Modify vendor/build output directories (`obj/`, `bin/`, `nuget/`).
- Log tokens, client secrets, or Authorization headers.
- Break backward compatibility without asking first (see Ask First) and getting explicit approval.

## Security Considerations

- **OAuth / OIDC SDK.** The Authentication API constructs authorization URLs but does not run the auth flow — callers must generate and validate `state` themselves (documented in `README.md`). ID-token validation lives in `src/Auth0.AuthenticationApi/Tokens/`.
- **Token handling.** The Management API `ManagementClient` wrapper acquires/refreshes tokens via `ITokenProvider`; never log tokens or secrets, and never send tokens to non-Auth0 hosts.
- **Telemetry.** Every request carries an `Auth0-Client` header (base64url-encoded `{name, version, env}`). It is on by default; preserve it when adding request paths.
- **Assemblies are strong-name signed** (`build/Auth0NetStrongName.snk`); `InternalsVisibleTo` is granted only to test assemblies with the public key. Never commit the key file's contents.
- **Secret scanning:** the repo runs Snyk (`.snyk`) and Semgrep (`.semgrepignore`); do not introduce hardcoded credentials.

## Testing

Unit tests need no credentials: `Auth0.Core.UnitTests` (xUnit) and `Auth0.ManagementApi.Test` (NUnit + WireMock.Net). The `*.IntegrationTests` projects hit a **live Auth0 tenant** and are Ask-First.

See [references/testing.md](references/testing.md) for frameworks, per-project commands, the live-tier requirements, and conventions. Read before writing or running tests.

## Code Style

C# with `Nullable` enabled repo-wide (`Auth0.Core` also sets `WarningsAsErrors=nullable`); PascalCase for public members, `_camelCase` for private fields; style enforced by `dotnet format` + `.editorconfig`. Generated ManagementApi code uses `Optional<T>` for PATCH semantics.

See [references/code-style.md](references/code-style.md) for naming rules, good/bad examples, and dominant patterns. Read when writing new types or converters.
